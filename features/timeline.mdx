---
title: "Timeline: daily summaries of your AI activity"
description: "The timeline turns your AI conversation history into daily AI-generated summaries with topics, tags, and linked conversations for each active day."
---

The timeline gives you a daily view of your AI conversation activity. Each entry is a structured summary generated by an LLM after your conversations for that day have synced. Instead of scrolling through individual conversations to reconstruct what you worked on, you can open the timeline and get an instant briefing on any given day — what topics came up, which platforms you used, and which conversations were involved.

## How timeline entries are generated

After your conversations sync, Memorybase queues a timeline generation job for the corresponding day. The backend LLM reads the day's conversations and produces a `summary` describing what you worked on, along with `tags` and `topics` that characterize the day's themes.

Generation is automatic — you don't need to trigger it manually. However, if an entry hasn't been generated yet, you can request it explicitly (see [Triggering generation manually](#triggering-generation-manually) below).

<Info>
  Timeline generation happens server-side, after conversations sync. On days where you had no conversations, no timeline entry is created.
</Info>

## What each timeline entry contains

A completed timeline entry includes:

| Field | Description |
|---|---|
| `date` | The calendar day the entry covers (e.g., `2026-02-25`). |
| `summary` | An AI-generated paragraph describing your activity for that day. |
| `tags` | Short keywords summarizing the day's topics. |
| `topics` | Broader theme categories for the day. |
| `tagMappings` | A map from tag names to the conversation IDs where each tag appeared. Useful for tracing which conversations drove a particular topic. |
| `conversations` | The list of conversations that contributed to the entry, each with its own `title`, `platform`, `tags`, `topics`, `summary`, and linked projects. |
| `status` | Processing state: `completed`, `processing`, or `not_generated`. |
| `processedAt` | When the LLM finished generating the entry. |

### Status states

<CardGroup cols={3}>
  <Card title="completed" icon="circle-check">
    The entry has been fully generated. `summary`, `tags`, `topics`, and `tagMappings` are populated.
  </Card>
  <Card title="processing" icon="spinner">
    Generation is in progress. The entry exists but its summary fields are not yet available.
  </Card>
  <Card title="not_generated" icon="clock">
    Conversations exist for this date but no generation has been attempted yet.
  </Card>
</CardGroup>

When you request an entry that is still `processing`, Memorybase returns the partial data immediately so you can display what's available while waiting for the full summary.

## Browsing your timeline

Open the Memorybase extension popup and navigate to the **Timeline** tab. Entries are displayed in reverse-chronological order, one per day.

The timeline uses cursor-based pagination with a `granularity` of `day`. Each page returns a batch of daily entries and a `nextCursor` — a date string in `YYYY-MM-DD` format — that you use to load the next page of older entries. When `nextCursor` is `null`, you've reached the beginning of your history.

<Tip>
  Expand any timeline entry to see the full list of conversations that contributed to it. Click a conversation to jump to its detail view.
</Tip>

## Viewing a specific date

To jump directly to a specific day, use the date picker in the timeline view or navigate to a date in `YYYY-MM-DD` format. Memorybase retrieves the entry for that exact date.

If no conversations exist for the requested date, the timeline returns a not-found response. If conversations exist but no entry has been generated yet, the response returns a placeholder with `status: not_generated`.

## Understanding tag mappings

The `tagMappings` field links each tag back to the specific conversations where it appeared:

```json
{
  "tagMappings": {
    "typescript": ["a1b2c3d4-e5f6-7890-abcd-ef1234567890"],
    "debugging": ["a1b2c3d4-e5f6-7890-abcd-ef1234567890", "b2c3d4e5-f6a7-8901-bcde-f12345678901"]
  }
}
```

In this example, the `typescript` tag appeared in one conversation, while `debugging` appeared in two. You can use this to trace which conversations drove a particular theme on a given day.

## Triggering generation manually

If a day's entry has `status: not_generated`, you can queue it for generation without waiting for automatic processing. Memorybase scans your history for any days that are missing timeline entries and queues them all at once.

You can also regenerate a specific existing entry — useful if you've synced new conversations for a day after the entry was already generated and want the summary to reflect the complete picture.

<Note>
  Regeneration replaces the existing `summary`, `tags`, `topics`, and `tagMappings` with newly generated values. The list of `conversations` linked to the entry is also updated to reflect any new additions.
</Note>

<AccordionGroup>
  <Accordion title="Why does my timeline entry show 'not_generated'?">
    This means Memorybase has your conversations for that day but hasn't generated the summary yet. This can happen if the automatic generation job hasn't run yet, or if you've added new conversations to a day that already had an entry. Use the manual trigger to queue generation immediately.
  </Accordion>
  <Accordion title="Can I edit a timeline summary?">
    Timeline summaries are AI-generated and read-only. If a summary is inaccurate, you can regenerate it, which re-runs the LLM over the day's conversations. If you need to add context the AI wouldn't have, add it via a project note instead.
  </Accordion>
  <Accordion title="Does archiving a conversation affect the timeline?">
    No. Conversations are linked to timeline entries at generation time. Archiving a conversation after the fact does not remove it from existing timeline entries. If you regenerate the entry, the archived conversation may still be included if it was active on that date.
  </Accordion>
</AccordionGroup>
